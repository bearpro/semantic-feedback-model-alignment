# Тезисы

Для ужесточения семантики кода на языке можно использовать
подходы обработки естественных языков на этапе статического анализа кода.

## Проблема

- Я пытаюсь решить задачу поиска противоречий в нормативных документов
- Для того чтобы находить противоречия мне нужен формальный способ записи нормативных положений
- Математический аппарат уже разработан: формализм DDL-LTLf (дефизибельная деонтическая логика с темпоральной семантикой), и показа алгоритм поиска противоречий. [Проказин, ИТ-Стандарт, 2026 год]
- Подход к программной реализации в целом понятный: специальный формальном язык промежуточного представления нормативных документов, наподобие ReqIF и других, но с учётом специфики моей задачи
- Ключевая проблема программной реализации, это межмодульное взаимодействие: два совершенно разных кода на специальном языке, написаны разными людьми (или LLM), в разных контекстах и в разное время, на основе разных исходных документов. А эти коды на формальном языке должны друг с другом корректно состыковаться, при чём даже не попарно, а на больших массивах документов.
- Основной подход к решению этой проблемы - межмодельное выравнивание - не рассматривается в этой статье.
- Но перед тем как приступать к межмодельному выравнивания есть небольшой способ упростить задачу: включить в правила языка дополнительную семантику, не входящую в формальную грамматику. Такая семантика могла бы принудить автора исходного кода особенным (и главное стабильным и предсказуемым) образом давать имена объектам исходного кода (полям, типам представляющих сущности, имена функций, и т.п.)
- Проблема такой идеи в том что её непонятно как энфорсить.

## Гипотеза

1. Стабильная заданная семантика неформальных объектов исходного кода на языке будет действительно полезна для этапа межмодельного выравнивания
2. Реализовать такую семантику можно в языковом сервере, и это получится удобно и эффективно использовать при работе с исходным кодом

## Эксперимент

- Несколько LLM моделей
- Несколько текстовых описаний реальных объектов
  - Текстовое описание API
  - Фрагмент стандарта на промышленное изделие
  - Какие-нибудь повседневные вещи (конкретный рецепт, описание ноутбука, инструкция по сбокре стола из IKEA)
- Единый базовый проипт "Опиши на C# модель, описывающую данную вещь"
  Варианты дополнения:
  - Ничего (контрольная группа)
  - Гайд по семантическому именованию параметров модели
  - Языковой сервер с обратной связью (дающий варнинги о неправильном именовании)
- Всё это закидываем в разные системы `model-aligment` и считаем скор.
